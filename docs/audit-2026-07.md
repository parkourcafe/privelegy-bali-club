## 2. Fixed in this branch (safe, no judgment call required)

1. **Canonical host unified to `www`.** Every runtime `https://otherbali.com` →
   `https://www.otherbali.com` (metadataBase, OG url, sitemap, robots sitemap,
   `lib/hub.ts` `SITE_ORIGIN`, `lib/guides.ts`, `lib/scenarios.ts`, `llms.txt`, all
   district/guide pages). Emails (`support@otherbali.com`) intentionally untouched.
2. **Baseline security headers** in `next.config.ts`: `X-Frame-Options: DENY`,
   `X-Content-Type-Options: nosniff`, `Referrer-Policy: strict-origin-when-cross-origin`,
   a restrictive `Permissions-Policy`, and `poweredByHeader: false`.
   - **CSP now shipped as Report-Only** (see item 10).
3. **Capacitor release logging** → `loggingBehavior: "none"`.
4. **Route JSON-LD** now maps venue `category` → the correct schema.org type
   (`CafeOrCoffeeShop` / `Restaurant` / `BarOrPub` / `HealthAndBeautyBusiness` /
   `SportsActivityLocation`), falling back to `LocalBusiness`.
5. **CI pipeline** (`.github/workflows/ci.yml`): lint → typecheck → build on PRs
   and pushes to `main`/`master`/`claude/**`. Added a `typecheck` script. First
   run green. This is the gate that would have caught the `/places` 500.
6. **Venue data validation at the trust boundary** (`lib/venue-validation.ts`,
   wired into `getPublishedVenues` + `getVenuesList`): structurally-broken DB
   rows (no slug/name/district, unknown category) are dropped from public lists
   and logged instead of crashing a sort or painting a dead card — the same
   class of bug as the `/places` null-address 500. Seed data is already
   guaranteed by the `Venue[]` type; the guard protects untyped DB rows. The
   category enum is kept in lockstep with `lib/types.ts` via a compile-time
   `satisfies` check. Content gaps (thin editorial) are still handled by the
   publication gate, not dropped.
   - **Follow-up:** a CI `data-validation` job that runs `venueContentIssues`
     against the **production** Supabase catalogue needs DB secrets in CI (a
     founder step) — deferred, noted here.
7. **Planner transparency — no silent defaults** (P1). The landing
   `DayIntentBuilder` pre-selected all 7 axes (`beach`/`view`/`couple`/`sunset`…)
   the traveller never chose and folded every one into the `/places` brief,
   silently biasing the shortlist. Now:
   - Every axis starts **unselected**; only tapped axes reach the brief
     (`buildHref` unchanged in precedence, but fed only real choices). A second
     tap clears an axis. The "map brief" line reflects only real choices; the
     CTA reads "Browse all places" until something is picked.
   - `/places` shows the live brief as **removable criteria chips** (every query
     token + district + category + mission + duration), each droppable, plus
     "Clear all".
   - Honest fallback copy when the full brief yields fewer than 3 / no ranked
     picks (instead of quietly showing less). The per-pick **"Matched because"**
     rationale already existed (`scoreVenue` reasons) and stays — now honest,
     since it can no longer cite an unchosen default.
   - Accessibility tie-ins: `aria-pressed` on all planner/place toggles
     (DayIntentBuilder, PlacesView + PlanView chips/moments) and an
     `aria-live="polite"` region on the `/places` result count.
   - **Follow-up:** unit tests over the ranking/brief combinations want a test
     runner; adding one changes the lockfile (would break `npm ci`), so it is
     deferred to a dedicated change rather than bundled here.
8. **Google Analytics disabled for the first build** (P0, founder decision
   2026-07-14: "GA off completely for the first build"). `components/Analytics.tsx`
   now loads GA4 only when production **and** `NEXT_PUBLIC_ENABLE_ANALYTICS === "1"`;
   the flag is unset, so no Google script is injected and no data reaches Google.
   `track()`'s GA leg no-ops when `window.gtag` is absent; the **first-party**
   event store (`/api/event` → Supabase) stays — it is how attribution / partner
   proof works, not third-party tracking. Privacy policy updated: explicit "no
   Google Analytics / no third-party or cross-app trackers" section, an
   email-based deletion path for device-linked events, and a refreshed date.
   With GA gone, the iOS manifest's `NSPrivacyTracking=false` is now clearly
   correct.
   - Superseded by item 9 (consent gate now built).
9. **Analytics consent gate + Privacy Choices + device forget** (P0). Behavioral
   analytics is now **opt-in**:
   - `lib/consent.ts` + `components/ConsentBanner.tsx` (first-visit Accept /
     Essential-only). `app/api/event/route.ts` enforces it **server-side** — no
     `bp_guest` minted, nothing logged, without `bp_consent=granted` — so every
     event caller is covered, not just `track()`. GA stays fully off.
   - Functional paths stay ungated: `/api/source` (explicit partner-link
     arrival) and `/api/redeem` (its own consent step). Partner-proof redemption
     attribution (guardrail #8) is therefore intact.
   - `/privacy/choices` (+ policy link): turn analytics on/off, **forget this
     device** (`/api/privacy/forget` clears `bp_guest`/`bp_consent` and calls the
     erasure RPC), and an email path for full deletion.
   - `supabase/migrations/0031_forget_guest.sql`: SECURITY DEFINER `forget_guest`
     RPC deletes behavioural events + saved places + shared lists + source, and
     **retains `redemption_events`** (billing/partner proof) — a retention choice
     documented in the migration for legal review, not decided silently.
     **Prod apply = founder step** (the app no-ops gracefully until then).
   - **Still open (founder + lawyer):** whether the functional `bp_guest` also
     needs consent; the full data inventory driving App Store privacy **labels**
     and `NSPrivacyCollectedDataTypes`; and whether erasure must extend to
     redemptions/consent. Not fabricated here.
10. **Content-Security-Policy — shipped Report-Only** (P1). `next.config.ts` adds
    a `Content-Security-Policy-Report-Only` header with an allowlist derived from
    the origins the browser actually touches (self + Next inline via
    `'unsafe-inline'`; `connect-src` self + `*.supabase.co` (browser Supabase
    client) + dormant GA; `img-src https:`; self-hosted fonts/video). Hardening
    directives: `object-src 'none'`, `base-uri 'self'`, `form-action 'self'`,
    `frame-ancestors 'none'`, `frame-src 'none'`, `upgrade-insecure-requests`.
    Verified: no iframes, no external client fetches beyond Supabase.
    - **Report-Only = zero breakage risk** — browsers never block, only report.
      Violations POST to `/api/csp-report` (logged) and show in the console.
    - **To enforce:** browse the site / watch the logs for a few days; when clean,
      rename the header key `Content-Security-Policy-Report-Only` →
      `Content-Security-Policy`. If prod build ever needs `'unsafe-eval'`
      (unusual for Next 16 App Router prod), a report will reveal it first.

> Note: a full `next build` / typecheck could not run in this environment
> (`node_modules` not installed — itself a symptom of the "no CI" finding). Changes are
> string-literal + standard-config only. **Run `npm ci && npm run build` before merge.**

## 3. Needs a founder / legal decision before I touch it

These are the audit's real P0s that I did **not** change unilaterally, because doing so
either alters product behavior, removes analytics the founder may rely on, or requires
information I don't have.

- **GA — RESOLVED 2026-07-14:** dropped for the first build (option a), see §2.8.
  Still open: the pre-consent **first-party** `bp_guest` identity + interaction
  events. Decide whether that needs a consent gate, and add an in-app / on-site
  **Privacy Choices** surface (reset id, request deletion — email deletion path
  now exists as an interim). Feeds the data inventory below.
- **Privacy policy + data inventory.** One inventory feeds four artifacts: web consent,
  policy text, App Store privacy labels, and `PrivacyInfo.xcprivacy`. Needs the real
  controller identity, processors (GA, Vercel, Supabase), retention, and a working
  server-side deletion path — lawyer review after the data flow is frozen. "Clear cookies
  to reset" is not a deletion mechanism while event rows persist in Supabase.
- **iOS: WebView → local app shell (Guideline 4.2 risk).** Removing `server.url` now would
  break the app because there is no bundled shell in `webDir: "ios-web"`. This is the
  multi-week native-value item: bundled navigation, saved places/routes, MapKit route line,
  offline cache, native share, universal links, WhatsApp-optional booking flow.
- **CI/e2e pipeline.** lint → typecheck → build → unit (ranking) → integration (prod-like
  schema) → Playwright core flows → link crawl → data-validation → preview smoke. This is
  the gate that would have caught the `/places` 500 before prod.
- **Second Vercel project** — archive or clearly label as staging (operational trap).
- **iOS toolchain** — migrate to Xcode 26 / iOS 26 SDK (required for submission), drop
  obsolete `armv7`, decide iPad support honestly.

## 4. Suggested order

1. **Stabilize** (mostly done here): canonical host ✅, security headers ✅, then CSP +
   data-validation on active venue rows + CI.
2. **Privacy alignment**: inventory → consent gate → Privacy Choices → policy rewrite →
   App Store labels → server-side deletion.
3. **Make the app an app**: local shell, saved routes, map, offline, deep links.
4. **Product polish**: remove hidden planner defaults (show removable criteria chips),
   "why this place" rationale, planner→detail links, taxonomy split (venue_type vs
   occasion), real photos + rights, hours + `verified_at`, numeric IDR price model.
5. **Submission**: TestFlight → device matrix → accessibility (VoiceOver/Dynamic Type) →
   metadata + real in-use screenshots → review notes.
